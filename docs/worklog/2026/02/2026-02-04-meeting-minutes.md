## 1. 公開アプリとカスタムアプリのデータアクセス制限

### 議論内容
- **現状**: 公開アプリでは、Shopify から許可が降りるまで顧客データ（顧客名・メールアドレス等）にアクセスできない制限がある。
- テスト用に**カスタムアプリ**でインストールすれば顧客情報を取得できる案もあったが、**採用しない**（福田さんのテストストア・シェアさんのストアでは顧客名等が出ているのは従来のカスタムアプリ利用時のみ）。
- 購入回数は Shopify 内部の顧客 ID で紐付けるよう修正済みのため、公開アプリでも取得できる想定。
- 公開アプリで顧客情報を取得するには審査が厳しく、保管方法・セキュリティ要件等の条件がある。許可後もインストール数増加等で見直し・取り消しの可能性あり（公式ドキュメントに記載）。

### 決定事項（最終）
- **テスト用カスタムアプリでインストールして顧客情報を取得する案はやめ、顧客名等はブランクで進める方針に最終決定**。
- 顧客名表示は「**項目は残したまま、空白の状態で進める**」。現段階では顧客名は必須ではなく、リスクを避けるため表示しない。後から申請を通して取得する運用も可能。

### アクション
- **福田**: Shopify サポートに、顧客情報アクセス申請のタイミング（アプリ申請と同時か、申請通過後か）を確認する。

## 2. データ表示の不具合（初回表示・複数ストア切り替え）

### 議論内容
- **事象**: 休眠顧客分析などで、初回アクセス時にデータが表示されず、ページをリロードすると表示される。
- 複数ストアを切り替えた際に、**ストア情報の切り替え・クリアがうまくいっておらず**、データ表示に不整合が発生する。
- フロントエンドでの「どのストアで動いているか」の管理と、切り替え時のクリア処理に問題がある可能性（推測）。

### 決定事項
- 複数ストアまわりの不具合として調査・修正する。

### アクション
- **福田**: 複数ストア切り替え時のデータ表示問題を調査・修正する（初期表示・ストア情報のクリア含む）。

## 3. 休眠顧客分析の計算ロジック

### 議論内容
- **不整合**: 左側の「総休眠顧客数」は**全顧客**を母数にしている一方、右側 2 つの指標は**購入履歴ありの顧客のみ**で算出しているため、休眠率が不正確（例: 顧客 100 名・総休眠 50 名で 54.9% → 購入履歴なし 9 名を除くと 50/91 が正しい等）。
- 購入履歴のない顧客は「休眠」に含めるべきではないという認識で一致。
- 過去に購入金額 0 円の注文を条件に含めていたため、購入回数のみで判定するよう修正済み（0 円注文はテストデータ等で存在し得るが、実運用では購入回数ベースで統一）。

### 決定事項
- **休眠顧客の計算は「購入履歴ありの顧客のみ」に統一する**。  
  総休眠顧客数も購入履歴あり顧客のみで算出し、表記・メッセージも全体で統一する。

### アクション
- **福田**: 休眠顧客率の計算ロジックを修正（購入履歴なし顧客を除外し、左側指標も購入履歴ありで統一）。

## 4. 顧客名表示の方針（動画・申請まわり）

### 議論内容
- 動画撮影時に顧客名が空白になる問題がある。
- 統計を見るだけなら顧客名は必須ではないという意見。リスクを考慮し、**現段階では顧客名を表示しない方向**で検討。
- 動画は「個人情報が載っていない状態で申請してよい」との記載を確認。申請は動画が撮れれば提出可能。

### 決定事項
- 顧客名の**列は残すが、値は空白のまま**で進める（レイアウトは一列減らさない）。  
  後から申請を通して取得し、表示を有効化する対応は可能。

## 6. データ検証とビジネス活用（シェアさんストア）

### 議論内容
- シェアさんのストアデータで、**「4 回購入した顧客が休眠顧客になっている」「5 回以上購入の顧客は休眠になっていない」**ことを検証したい。
- オーダーテーブル等で「5 回以上購入している人が休眠になっていない」ことを確認できれば、DM 施策の提案材料・営業トークに使える。
- データ同期は福田さん側で実施してよい（鳥谷さんには福田さんから連絡する）。

### 決定事項
- シェアさんのストアで**データ同期を実施**し、DB（オーダーテーブル等）で実データを確認する。  
  成功事例として DM アプリ・EC レンジャー両方の営業に活用する。

### アクション
- **福田**: シェアさんのストアでデータ同期を実施し、整合性・休眠判定を確認する。
- **濱地**: DM 施策の提案資料を作成する（LTV ラボ等で手が塞がっているため、ポンチ絵でも可・AI 活用も検討）。

## 7. UI/UX 改善・不具合（Backlog 共有内容との対応）

### 議論内容（Backlog チケット・会議での追加指摘を反映）
- **リスクレベル**: ヘルプ（?）で表示されるラベル色と、実際のリストのラベル色が不一致 → 修正する。
- **商品カテゴリー**: 全て「未分類」になっている不具合。  
  商品タイプは取得できているが、カテゴリーは API レスポンスにらしきものが見つかっておらず、取得方法含め調査が必要。
- **CSV**: 金額のカンマでセルが分割される問題。
- **Excel**: 拡張子が `.excel` になっており、Excel として認識されない。
- **操作の統一**: 並び順変更時は即時再描画、表示データ変更時は「分析実行」を押す必要があるなど、**条件変更時の挙動が画面ごとにバラバラ**で分かりづらい。対象年変更時はデータ再取得が必要なのに、タイトルのみ切り替わる問題もあり。
- **前年比**: 常に +0.0% になっており、比較対象も不明。固定値で表示している可能性があり、修正または削除を検討。
- **構成比**: パーセント表示に「+」は不要では、という指摘（% のみでよい）。

### 決定事項
- 上記を Backlog に反映し、**4 番として「2月○日ミーティングでの追加指摘事項」をまとめて登録**する（表題で 4 番を追加。既存 3 番までは維持）。
- 対応優先度・工数は未確定だが、強めに作業する。

### アクション
- **福田**: リスクレベルヘルプのラベル色を実際の表示に合わせて修正する。
- **福田**: 商品カテゴリーのデータ取得を調査・修正する（API ・レスポンスの確認含む）。
- **福田**: Backlog 4 番として、本ミーティングでの追加指摘事項をまとめて登録する。
- **福田**: UI/UX の各種不具合を修正する（CSV カンマ、Excel 拡張子、前年比表示、構成比の「+」、条件変更時の挙動統一など）。  
  （本文を直接触る場合あり・小野さんから）

## 10. 関連 Backlog チケット（要約）

福田さんあてに共有された不具合報告・確認希望点の要点は以下のとおり。

- **前年同月比分析**: 商品カテゴリーが未分類、スクロール、CSV/Excel、対象年変更時のデータ不更新、表示データ変更時のみリセットされる挙動の仕様確認、成長率バッジのホバー視認性。
- **購入回数分析**: 前年比が常に +0.0%、構成比の「+」表記、リスク評価のレイアウト確認。
- **休眠顧客分析**: 表示不安定・初回で出ない・リロードで出る、顧客 ID 以外が取れない場合、顧客名非表示、CSV の推奨アクション削除希望、CSV のユーザー情報未取得、リスクレベルヘルプとラベル色の不一致、顧客 ID のスタイル・参照元の説明希望。

上記は本議事録の「7. UI/UX 改善・不具合」および各セクションと対応している。
